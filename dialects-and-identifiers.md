<a name="introduction"></a>
## Introduction

A *dialect* tells DataFlow three things: how to quote identifiers, which identifier shapes
to accept, and how to render pagination. DataFlow ships seven dialects under
`com.binwi.dataflow.dialect`. Picking the right one lets you target case-sensitive schemas
and database-specific pagination without writing raw SQL.

The default is `GenericDialect` — the strictest and most portable option, and fully
backward compatible with pre-dialect code.

<a name="the-built-in-dialects"></a>
## The Built-in Dialects

| Dialect | Quote | Identifier pattern | Pagination |
|---|---|---|---|
| `GenericDialect` (default) | none (bare) | `^[a-z][a-z0-9_]*$` | `LIMIT ... OFFSET ...` |
| `PostgresDialect` | `"col"` | `^[A-Za-z_][A-Za-z0-9_]*$` | `LIMIT ... OFFSET ...` |
| `H2Dialect` | `"col"` | `^[A-Za-z_][A-Za-z0-9_]*$` | `LIMIT ... OFFSET ...` |
| `MysqlDialect` | `` `col` `` | `^[A-Za-z_][A-Za-z0-9_]*$` | `LIMIT ... OFFSET ...` |
| `MariaDbDialect` | `` `col` `` | `^[A-Za-z_][A-Za-z0-9_]*$` | `LIMIT ... OFFSET ...` |
| `OracleDialect` | `"col"` | `^[A-Za-z_][A-Za-z0-9_]*$` | `OFFSET ... ROWS FETCH NEXT ... ROWS ONLY` |
| `SqlServerDialect` | `[col]` | `^[A-Za-z_][A-Za-z0-9_]*$` | `OFFSET ... ROWS FETCH NEXT ... ROWS ONLY` |

Each dialect is a singleton, accessed through its `INSTANCE` field:

```java
DB db = new DB(jdbc, PostgresDialect.INSTANCE);
```

Under Spring Boot, select the dialect with the `binwi.dataflow.dialect` property instead —
see [Spring Boot Integration](spring-boot-integration.md#configuring-the-dialect).

<a name="identifier-validation"></a>
## Identifier Validation

Every table, column, and alias you pass to the public API is validated *before* it reaches
the database. This is what makes SQL injection through identifiers structurally impossible —
an invalid name never becomes part of a statement.

- Under `GenericDialect`, identifiers must match `^[a-z][a-z0-9_]*$` — lowercase
  snake_case, starting with a letter.
- Under any other dialect, validation relaxes to `^[A-Za-z_][A-Za-z0-9_]*$`, so camelCase
  and PascalCase identifiers pass through.

A name that fails validation throws an `InvalidIdentifierException`:

```java
db.table("users; DROP TABLE users");   // throws InvalidIdentifierException
db.table("users").select("1=1");       // throws InvalidIdentifierException
```

You can check a name yourself with the static helpers:

```java
DB.isValidName("user_id");        // true
DB.isValidName("user id");        // false
DB.isValidAliasName("displayName"); // alias-shaped check
```

<a name="qualified-columns-and-aliases"></a>
## Qualified Columns and Aliases

Columns may be *qualified* with a single dot to reference a table, and aliased with `as`:

```java
db.table("users")
    .select("id", "users.name as displayName")
    .get();
```

DataFlow quotes each part of a qualified name independently (`"users"."name"`) and quotes
the alias separately. Both halves are validated against the active dialect's identifier
pattern.

<a name="the-generic-dialect"></a>
## The Generic Dialect

`GenericDialect` is the default and the most conservative choice. It:

- Emits **bare** identifiers with no quoting.
- Accepts only lowercase snake_case (`^[a-z][a-z0-9_]*$`).
- Uses standard `LIMIT ... OFFSET ...` pagination.

This shape is portable across virtually every database and is the safest default for
greenfield schemas that you control. If your schema is all lowercase snake_case — the
common convention — you may never need another dialect.

<a name="case-sensitive-schemas"></a>
## Case-Sensitive Schemas

Some schemas use mixed-case identifiers — a quoted Postgres table, or a schema generated by
Hibernate with camelCase column names. For these, choose the matching dialect. Doing so:

- **Relaxes validation** to `[A-Za-z_][A-Za-z0-9_]*`, so mixed-case names are accepted.
- **Quotes every emitted identifier** with the dialect's native delimiter, so case-sensitive
  names resolve correctly.
- **Lowercases internal placeholder names**, so generated SQL always survives Spring's
  named-parameter parser. For example, `where("firstName", "Eve")` binds `:firstname_1`
  even though the column keeps its original case in the SQL.

```java
DB db = new DB(jdbc, PostgresDialect.INSTANCE);

db.table("Users")
    .where("firstName", "Eve")
    .get();
// SELECT * FROM "Users" WHERE "firstName" = :firstname_1
```

<a name="pagination-by-dialect"></a>
## Pagination by Dialect

`take()` and `skip()` render differently depending on the dialect:

- **Postgres, MySQL, MariaDB, H2** use `LIMIT :_limit OFFSET :_offset`.
- **Oracle and SQL Server** use `OFFSET :_offset ROWS FETCH NEXT :_limit ROWS ONLY`.

Both `LIMIT`/`OFFSET` values are bound as named parameters (`:_limit`, `:_offset`) — never
concatenated into the SQL.

> [!WARNING]
> SQL Server requires an `ORDER BY` whenever `take()` or `skip()` is used, because
> `OFFSET ... FETCH` is only valid alongside an ordering. Always add `orderByAsc(...)` /
> `orderByDesc(...)` before paginating on SQL Server.

SQLite and pre-12c Oracle are not supported out of the box. If you must target them, reach
for the raw escape hatches or `unsafeQuery` — see
[Raw SQL Escape Hatches](raw-sql-escape-hatches.md).

<a name="a-worked-example"></a>
## A Worked Example

Here is a Postgres-dialect query showing quoting, joins, and pagination together:

```java
DB db = new DB(jdbc, PostgresDialect.INSTANCE);

db.table("Users")
    .select("id", "users.name as displayName")
    .where("active", true)
    .innerJoin("posts", "posts.user_id", "users.id", on -> {})
    .orderByAsc("createdAt")
    .take(10)
    .get();
```

Renders:

```sql
SELECT "id", "users"."name" as "displayName" FROM "Users"
  INNER JOIN "posts" ON "posts"."user_id" = "users"."id"
  WHERE "active" = :active_1
  ORDER BY "createdAt" ASC LIMIT :_limit
```

Notice that every identifier is quoted, the alias is quoted independently, and the limit is
a bound parameter — all driven by the dialect you selected.
